# ドキュメント作成方針 - 背景資料

本資料は、[doc-policy.mdc](../rules/doc-policy.mdc)のルールを策定するに至った経緯、背景、実例を説明するものです。

---

## 1. [原則としてマークダウン形式（.md）でドキュメントを作成してください](../rules/doc-policy.mdc)

### 背景・理由

プロジェクトにおけるドキュメント管理では、以下の観点から一貫したファイル形式の採用が重要です：

- **バージョン管理との親和性**: Gitなどのバージョン管理システムで差分を明確に確認できる
- **テキストベースの利便性**: 特定のソフトウェアに依存せず、あらゆる環境で編集・閲覧が可能
- **検索性の向上**: テキストファイルであるため、grepやIDEの検索機能が活用しやすい
- **軽量性**: バイナリファイルと比較してファイルサイズが小さく、リポジトリの肥大化を防げる

### 実例

- 技術資料: `README.md`、`API_SPECIFICATION.md`
- 議事録: `meeting_YYYYMMDD.md`
- 報告書: `poc_report.md`

## 2. [文体は「です・ます調」を基本とし、技術仕様や論理的考察など客観性が求められる箇所では「だ・である調」の使用を許容します](../rules/doc-policy.mdc)

### 背景・理由

**基本姿勢として「です・ます調」を採用する理由：**
- 読み手に対して丁寧で分かりやすい印象を与える
- 社内外問わず、幅広い読者層に適応できる
- プロジェクトドキュメントとしての信頼性を高める

**「だ・である調」を部分的に許容する理由：**
- 技術仕様の定義や論理的考察においては、簡潔さと客観性が優先される
- 専門的な内容では「だ・である調」のほうが読みやすい場合がある
- ただし、文章の一貫性を保つため、段落内での混在は厳禁

### 実例

**です・ます調の使用例（基本）:**
```markdown
本システムは、ユーザーの入力を受け取り、適切な処理を実行します。
処理結果はデータベースに保存され、画面に表示されます。
```

**だ・である調の使用例（技術仕様）:**
```markdown
### システム要件
- 対応OS: Linux、Windows、macOS
- 必要メモリ: 最小4GB、推奨8GB以上
- データベース: PostgreSQL 12以上である
```

## 3. [ドキュメントの冒頭で、目的と読者が得られる成果を明確に記述してください](../rules/doc-policy.mdc)

### 背景・理由

ドキュメントを読む前に「このドキュメントから何が得られるか」を明確にすることで：
- 読者は自分にとって必要な情報かどうかを素早く判断できる
- 読むべき箇所を絞り込める（時間の節約）
- ドキュメント作成者自身も、執筆の方向性を明確に保てる

### 実例

```markdown
## この資料のゴール

本資料は、〇〇システムのAPI仕様を定義し、フロントエンド開発者が迷わずに実装を進められる状態になることを目的としています。

**読了後に得られること:**
- 各APIエンドポイントの詳細仕様
- リクエスト・レスポンスのデータ構造
- エラーハンドリングの方法
```

## 4. [結論から書くPREP法（Point→Reason→Example→Point）を用いて構成してください](../rules/doc-policy.mdc)

### 背景・理由

PREP法を用いることで、ドキュメントの可読性と理解度が大幅に向上します：

- **P (Point - 結論)**: 最初に結論を述べることで、読者は何について書かれているかを即座に把握できる
- **R (Reason - 理由)**: なぜその結論に至ったのかを説明し、論理的な根拠を示す
- **E (Example - 具体例)**: 実例を示すことで、抽象的な概念を具体的に理解させる
- **P (Point - 結論の再確認)**: 最後に結論を再度述べることで、内容を強調し記憶に定着させる

### 実例

```markdown
## データベース接続にはコネクションプーリングを使用します (P)

### 理由 (R)
コネクションプーリングを使用することで、データベース接続の確立・切断のオーバーヘッドを削減し、システム全体のパフォーマンスを向上させることができます。

### 具体例 (E)
以下のように設定ファイルでプール設定を行います：

\`\`\`yaml
database:
  pool_size: 10
  max_overflow: 20
\`\`\`

### まとめ (P)
したがって、本プロジェクトではコネクションプーリングを標準として採用します。
```

## 5. [事実と意見を明確に区別し、客観的な記述を徹底してください](../rules/doc-policy.mdc)

### 背景・理由

技術ドキュメントにおいては、読者が情報の信頼性を判断できることが重要です：

- **事実**: データ、実測値、確認された動作など、客観的に検証可能な情報
- **意見・推測**: 分析、予測、提案など、主観的な判断を含む情報

これらを明確に区別することで：
- 読者は情報の確実性を正しく評価できる
- 誤解や不正確な伝達を防げる
- ドキュメントの信頼性が向上する

### 実例

**❌ 悪い例（事実と意見が混在）:**
```markdown
このシステムはたぶん高速に動作します。エラーはほとんど出ないと思います。
```

**✅ 良い例（事実と推測を区別）:**
```markdown
このシステムのレスポンスタイムは平均150msです（実測値）。
エラー率は過去30日間で0.01%でした（ログデータに基づく）。
今後もこの水準を維持できると考えられますが、負荷増加時の影響については継続的な監視が必要です。
```

## 6. [人名を記載する際、プロジェクトメンバー（本間、寺本など）には敬称をつけず、名字のみで記載してください。ただし、お客様については「様」をつけてください](../rules/doc-policy.mdc)

### 背景・理由

**プロジェクトメンバーの敬称を省略する理由：**
- **簡潔性**: 文書の可読性を高め、冗長性を排除する
- **一貫性**: メンバー間で敬称の有無が混在することを避ける
- **効率性**: 記述の手間を削減し、文書作成を効率化する
- **プロフェッショナリズム**: ビジネス文書として適切な形式を維持する

**お客様には必ず「様」をつける理由：**
- **ビジネスマナー**: 顧客に対する敬意と配慮を示す
- **信頼関係の構築**: 丁寧な対応が顧客との良好な関係を維持する
- **社外向け**: お客様への言及は社外向けの意味合いを持つため、敬称が必須

### 実例

**プロジェクト内部文書（基本パターン）:**
```markdown
## 担当者
- 要件定義: {メンバー名1}
- 開発: {メンバー名2}
- テスト: {メンバー名2}

## 打ち合わせ参加者
- {メンバー名1}、{メンバー名2}
- {お客様名}様（お客様）
```

**会議議事録:**
```markdown
## 参加者
- {メンバー名1}、{メンバー名2}
- {組織名}: {お客様名1}様、{お客様名2}様

## 議事内容
{お客様名}様から要件について以下のご指摘をいただきました。
{メンバー名1}が対応方針を説明し、{メンバー名2}が技術的な実現可能性を確認しました。
```

**報告書:**
```markdown
## プロジェクト体制
- プロジェクトマネージャー: {メンバー名1}
- 開発リーダー: {メンバー名2}
- {組織名}: {お客様名}様
```
